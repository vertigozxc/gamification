# Working Style & Rules

- User values root-cause fixes, decisive execution, and avoiding repeated passes over the same UI/bug area.
- Prefer identifying the true rendering/ownership layer first, then make one focused fix and verify it.
- **Mobile-first**: user builds mobile app primarily (real iPhone via Xcode + TestFlight, Render-hosted backend). Validate every fix against mobile flow, not desktop assumptions.
- When making any design elements, take into account mobile-first approach with best UI/UX design practices and Apple Human Interface Guidelines.
- All changes, improvements, or new functionality must be created immediately for the different languages available in web and mobile applications, as well as for the interface themes available in web and mobile applications.
- **i18n is mandatory for every new visible string.** Any text I add to the UI — labels, headings, buttons, tooltips, error messages, placeholders, notifications, numeric suffixes, everything — must be added to BOTH `en` and `ru` locale sections of `client/src/i18nConfig.js` in the same change. Hardcoded English-only (or Russian-only) strings in JSX are forbidden. If a string is dynamic (counts, names), build it from translated templates (e.g. `"Level {n}"`) and localise both the template and any punctuation.
- **Theme support is mandatory for every new UI surface.** All 3 interface themes — `adventure`, `balance`, `light` — must render the new UI correctly. Use theme variables (`var(--color-primary)`, `var(--panel-bg)`, `var(--panel-border)`, `var(--color-text)`, `var(--color-muted)`, etc.) or the `themeId` / `useTheme()` hook — never hardcode colors like `#111` or `white` for chrome/backgrounds. Test visually in each theme before declaring a feature done.
- **Exception — procedural SVG city content.** The isometric city, the 2D district scenes, the mountains/sky landscape inside `CityIsometricOverview.jsx` and `DistrictView.jsx` are intentionally NOT theme-driven; their look is cohesive across themes and switching UI theme should not change how the city renders. Only the surrounding interface chrome (cards, buttons, panels, text) follows the 3-theme rule.
- Remember that we have two applications, web and mobile — the mobile one should work as a priority and all changes are tested and applied on it.
- **Debug via logs, never guess**: when a bug isn't immediately obvious from code, add visible diagnostic output FIRST (status codes, per-step logs, banners on screen) and ship that. Only fix once real data shows the failing step. Avoid speculative fixes — they waste deploy cycles and erode trust.
- Diagnostic patterns that worked: visible banner overlay above WebView showing real-time auth status; per-step fetch status logging directly in callback HTML body.
- Always verify state on the server before signaling success to client (e.g. bridge-check after store).
- Reproduce 500/4xx errors with curl directly against the deployed endpoint before assuming code logic is wrong — could be CORS/middleware throwing.
- **Production URL map — do not guess.** Render service names and their public URLs diverge for this project. Canonical mapping (see `render.yaml`, `infra/cloudflare/worker.js`, `README.md` "Multi-Region API"):
  - `life-rpg-api-us` service (Oregon) → **`https://life-rpg-api.onrender.com`** (NOT `life-rpg-api-us.onrender.com` — that subdomain returns 404 and is a wrong guess).
  - `life-rpg-api-eu` service (Frankfurt) → **`https://life-rpg-api-eu.onrender.com`**.
  - Client and mobile talk to the Cloudflare Worker router at `https://life-rpg-api-router.evgeny-mahnach.workers.dev` (and the custom `https://api.life-rpg.app`), which fans out to the two origins above.
  - Before concluding "region X is stale", curl BOTH canonical URLs (`/healthz`) — never rely on the service name as a hostname.
- **NEVER delete, revert, reset, overwrite, or otherwise destroy uncommitted work, files, database rows, or any other data without EXPLICIT user approval in the current turn.** No `git checkout --`, `git reset`, `rm`, `DROP`, `Write` over an existing file, etc., as a silent cleanup step. If a rollback or cleanup seems useful, ASK first with specifics (which files, which lines) and wait for the user to say yes. Even when the user sounds frustrated or asks to stop, stop in place — do not retroactively "clean up" by throwing away work.
- **NEVER push to `main` (or merge to `main`) unless the user writes the literal word "ПУШ" in the current turn.** Pushing to feature branches is fine with a normal "save/commit" instruction, but `main` auto-deploys to Render prod — treat it as protected. No creating PRs that target `main` for auto-merge, no `git push origin main`, no `gh pr merge` without the explicit "ПУШ" keyword from the user.
- **Every interactive button must have visible press feedback.** Always attach the `mobile-pressable` class to any `<button>` we add (gives `:active` scale + brightness + tap-highlight reset). For high-emphasis or theme-coloured controls (purchases, primary CTAs, steppers, tier dots, filter pills), pair it with a component-specific `:active:not(:disabled)` rule in `client/src/styles.css` that visibly changes background / border / shadow — use `color-mix(in srgb, var(--color-primary) …%, transparent)` so the flash follows the active theme. Plain `<button>` with no press feedback is forbidden in any new UI.
- **Selected / active states use theme-primary tint, not hardcoded amber.** For "this card/row is selected" backgrounds prefer `color-mix(in srgb, var(--color-primary) ~12%, transparent)` (or `var(--color-accent)` when the design calls for the accent family). Hardcoded colours like `rgba(250, 204, 21, …)` or `var(--color-accent-dim)` at >20% alpha read wrong in the balance (green) and light (pink) themes — stay with `color-mix` on the primary/accent var so the tint follows the active theme automatically.
- **Icon style: Apple / iOS / SF Symbols aesthetic.** New icons must look like they belong next to an SF Symbol: single-weight outline strokes (roughly 1.6–2.0 px at 24 px viewBox), rounded line caps and joins (`strokeLinecap="round"` + `strokeLinejoin="round"`), geometrically simple shapes, no gradients or drop shadows on the glyph itself (the surrounding chip can glow — the glyph can't). Prefer `stroke="currentColor"` so the icon picks up its container's text colour and themes cleanly. Avoid Material-style filled icons, dual-tone glyphs, or skeuomorphic/3-D looks. When an emoji would work (🔒, ❄️, 🪙, 🏖️ etc.), emoji is fine too — just don't mix an emoji with an ad-hoc SVG in the same surface. If you're unsure, grep Lucide or SF Symbols for a canonical shape and mirror its proportions.
